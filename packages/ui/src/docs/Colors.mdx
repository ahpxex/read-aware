# Colors

ReadAware's palette is built from **semantic, theme-adaptive tokens** defined in
the `@theme` block of `apps/web/src/index.css`. Components reference the semantic
tokens (`text-fg`, `bg-surface`, `border-border-strong`, …) instead of raw
`stone-*` shades, so every surface inverts automatically when
`html[data-theme="dark"]` swaps the token values.

## Semantic tokens

| Token | Utility | Light | Dark | Role |
|-------|---------|-------|------|------|
| `--color-fg` | `text-fg` | `#1c1917` | `#f5f5f4` | Primary text, solid button fill |
| `--color-fg-muted` | `text-fg-muted` | `#57534e` | `#a8a29e` | Secondary text, labels, captions |
| `--color-fg-subtle` | `text-fg-subtle` | `#a8a29e` | `#78716c` | Tertiary text, placeholders |
| `--color-inverse-fg` | `text-inverse-fg` | `#fafaf9` | `#1c1917` | Text on `fg`-colored fills |
| `--color-surface` | `bg-surface` | `#ffffff` | `#322d29` | Raised surfaces (cards, popovers) |
| `--color-fill` | `bg-fill` | `#f5f5f4` | `#292524` | Quiet fills, hover washes |
| `--color-fill-strong` | `bg-fill-strong` | `#e7e5e4` | `#44403c` | Firmer fills (kbd, inline code) |
| `--color-border` | `border-border` | `rgb(28 25 23 / 0.1)` | `rgb(231 229 228 / 0.12)` | Hairline dividers |
| `--color-border-strong` | `border-border-strong` | `#d6d3d1` | `rgb(231 229 228 / 0.18)` | Firmer borders (inputs, emphasis) |
| `--color-paper` | `bg-paper` | `#f5f1e8` | `#1c1917` | App canvas |
| `--color-paper-warm` | `bg-paper-warm` | `#eee8d5` | `#292524` | Warm canvas variation |

The swatches above are painted with the live `var(--color-*)` values -- flip the
theme toolbar control and they follow.

## Dark theme

`html[data-theme="dark"]` (set by the in-app appearance control, not the OS media
query) overrides every token above. There is no second component API for dark
mode: a component written against the semantic tokens is dark-ready by
construction. **Never hardcode raw `stone-*` classes in components or story
harnesses** -- they do not invert and break one of the two themes.

## Functional accents

Two colors exist outside the stone range, both strictly functional:

- **Emerald** for success (`Alert` / `Toast` `success` variants).
- **Red** for the destructive `Button` `danger` variant.

Errors themselves are not red: the `destructive` variants of `Alert` and `Toast`
stay in the stone palette (firmer border, full-value title). No other accent
color may be introduced without a state to justify it.

## Verifying

The **Design System/Guidelines/Theme Audit** story renders every control on one
token-driven canvas. Use it (with the theme toolbar toggle) to confirm nothing
goes dark-on-dark or light-on-light.
